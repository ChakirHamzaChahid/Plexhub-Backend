## backend-developer — Wave 3 (backfill OMDb manuel, `/refacto` dual-provider enrichment)

Endpoint admin manuel de backfill des notes OMDb (`imdb_rating`/`imdb_votes` + `display_rating`
blend), Wave 3 de `docs/plans/2026-07-20-omdb-rating-enrichment-design.md` §C4, construit
uniquement sur les primitives partagées déjà mergées de la Wave 1 (`app/utils/rating_blend.py`,
`omdb_service.get_by_imdb_id`/`is_configured`/`get_request_count`, `omdb_scrape_cache_service`) —
`enrichment_worker.py` non touché (propriété Wave 2/sync-specialist).

**Créés** : `app/workers/enrichment_backfill_worker.py` (miroir du précédent `embedding_worker`
202-job in-memory — `OrderedDict` job store, `JOBS_CAP`, `create_background_task` — plus un garde
mono-run en mémoire `_running`). Phase A pagine par keyset `(server_id, rating_key)` les `media`
incomplets (`imdb_id` posé, `imdb_rating` NULL, `type IN (movie, show)` selon `mediaType`),
résout OMDb cache-first/budget-gated (fail-open intégral : non configuré / budget épuisé /
not-found / exception → skip, jamais de crash), écrit `imdb_rating`/`imdb_votes` en
COALESCE fill-missing + `display_rating` via `blend_display_rating_case`. Phase B (optionnelle,
`recomputeDisplayRating=true` par défaut) exécute `recompute_display_rating_stmt()` (SQL seul,
idempotent) pour guérir les lignes déjà complètes mais dont le rating a dérivé (clobber
`content_hash` côté sync). `app/api/enrichment.py` (Pattern C, self-prefixé
`/api/admin/enrichment`, `dependencies=[Depends(verify_master_key)]` module-level — secret maître
seul, comme les 2 miroirs JSON download) : `POST /omdb-backfill` → 202 + job (409 si un backfill
tourne déjà), `GET /jobs/{jobId}` → 200 | 404. Schémas Pydantic v2 camelCase
(`OmdbBackfillRequest`/`OmdbBackfillJobResponse`, `to_camel`+`populate_by_name`), aucun dict brut.

**Modifié** : `app/main.py` (import + `app.include_router(enrichment.router)` à côté de
`downloads.router`/`plex_downloads.router`, même convention Pattern C — pas de garde ajouté au
`_guard` list, le routeur s'auto-garde). Aucune migration (toutes les colonnes existent déjà
depuis la Wave 1/`omdb_scrape_cache` migration 022).

**Bug trouvé + corrigé en cours de dev** : la première version de `_run_phase_a` ouvrait une
session imbriquée (`_fetch_omdb_by_id`'s cache-read) PENDANT que la session d'écriture de la page
était encore ouverte/non-committée — sur un pool à connexion unique (`StaticPool`, notamment
`sqlite:///:memory:` en tests), la session imbriquée partage la même connexion DBAPI et sa
fermeture implicite annule silencieusement le travail non-committé de la session d'écriture (1ʳᵉ
ligne d'une page perdue, la 2ᵉ survivait). Fix : séparation stricte fetch (toutes les résolutions
OMDb d'une page, aucune session d'écriture ouverte) puis write (une seule session, tous les writes
de la page, commits batchés) — même discipline fetch-puis-apply que `enrichment_worker`. Capturé
par `tests/test_enrichment_backfill.py::TestPhaseASelection::test_only_eligible_rows_get_filled`.

**Tests** : `tests/test_enrichment_backfill.py` créé (21 tests) — auth (401 sans clé/mauvaise
