# CLAUDE.md — PlexHub Backend

## §1. Résumé
Backend **FastAPI** (Python, async) du stack PlexHub : miroir de bibliothèques **Xtream-IPTV** (VOD + séries + chaînes Live + EPG), enrichissement **TMDB**, validation de flux, génération de **bibliothèque Plex/Jellyfin** (NFO + arborescence + .strm/images), **recommandations IA** (embeddings + recherche vectorielle), **génération LLM** (Ollama/gemma4 : présentations + chat) et **appairage TV** device-flow. Consommé par l'app Android `PlexHubTV`. Le backend ne stocke **aucun historique utilisateur côté IA** : `ai_tmdb_cache` + `ai_embeddings` sont clés sur `tmdb_id`. Cible de déploiement **Docker/Linux** : l'élection master-worker s'appuie sur `fcntl.flock` (POSIX, `app/main.py:226-227`).

## §5. Flux clés (bout-en-bout)
- **§5.1 Sync** — `sync_worker.run_all_accounts()` → `sync_account(id)` (lock par compte) → `xtream_service` (catégories, VOD, séries+épisodes, Live, EPG) → upsert incrémental par `dto_hash`/`content_hash` + `differential_cleanup*`. Planifié toutes `SYNC_INTERVAL_HOURS` (défaut 6 h). Réf : `workers/sync_worker.py:854,1290`.
- **§5.2 Enrichissement (scraping TMDB)** — au sync, les titres Xtream sont nettoyés par **`clean_title`** (préfixes langue/pays empilés, séparateurs scène `.`/`_`, année parenthésée OU nue, tags qualité partout, crochets). `enrichment_worker.run()` (Phase 1 movies, Phase 2 séries) résout chaque item :
  - **Cache de scrape persistant d'abord** (`scrape_cache_service` + table `tmdb_scrape_cache`, clé `media_type|titre_normalisé|année`) → **0 appel TMDB** pour un même film/série (cross-comptes + survit au redémarrage ; TTL 30 j match / 3 j négatif).
  - Sur miss : **matching façon ScraperMatcher** (`tmdb_service._best_match`) — `titleScore = max(sim(titre), sim(original_title))` (rapidfuzz `max(ratio, token_set_ratio)`), `yearScore` 1.0/0.8/0.5/**0.0** (mismatch = garde-fou homonyme), `confidence = 0.7·title + 0.3·year`, tri `confidence`>`vote_count`. **Auto-match** si `confidence≥0.85` ET `titleScore≥0.90` ET **marge≥0.05** vs 2ᵉ.
  - **Désambiguïsation par résumé** : si doute (marge < 0.05), on départage avec le **résumé Xtream** (`existing_summary`) vs `overview` candidat (`token_set_ratio`, seuils `SUMMARY_MIN_SIM=0.30` / `SUMMARY_TIEBREAK_MARGIN=0.10`).
  - **Fallback** : défaut → sans année → `language=en-US` → `/search/multi`. Le meilleur score est enregistré (`tmdb_match_confidence`) même sans match. Métrique `plexhub_tmdb_match_total{media_type,result}`.
  - Borné par `ENRICHMENT_DAILY_LIMIT` (défaut **50000**, appels API réels), concurrence 8, `MAX_ATTEMPTS=3`. Réf : `workers/enrichment_worker.py`, `services/tmdb_service.py`, `services/scrape_cache_service.py`, `utils/string_normalizer.py`.
- **§5.3 Validation de flux** — `health_check_worker.run_pipeline_validation()` (pipeline) et `run()` (cron) : HEAD puis Range GET (magic bytes), concurrence `STREAM_VALIDATION_CONCURRENCY` (défaut 20), marque cassé après `STREAM_BROKEN_THRESHOLD` (défaut 3) échecs ou échec **définitif** (404/403/error-CT/empty/magic-fail), re-check toutes `STREAM_VALIDATION_RECHECK_HOURS` (défaut 24), **circuit breaker** par compte à 90 % d'échecs. Réf : `workers/health_check_worker.py:274,89-103`.
- **§5.4 Dédup unifiée (DEUX consommateurs, logique partagée)** — la dédup par `unification_id` (imdb→tmdb→titre+année) vit dans **`aggregation_service`** et alimente **les deux** chemins. ## §9. Pièges (house law — à respecter impérativement)
1. **Cold start IA ~30 s** au 1ᵉʳ `/rank` : fastembed télécharge + charge les poids ONNX (cold start documenté `embedding_service.py:11`). Les appels suivants sont rapides (singleton `_model`, lock `_MODEL_LOCK`, `embedding_service.py:71-82`). Override via `AI_EMBED_MODEL` ; cache persistant via `AI_EMBED_CACHE_DIR`.
2. **3 motifs de 503 IA vectorielle** (contractuels) : `AI service not configured` (`AI_API_KEY` vide, `deps.py:32-36`) · `AI vector storage unavailable` (sqlite-vec non chargé, `deps.py:37-41`) · `AI model unavailable` (fastembed KO → `EmbeddingUnavailableError`, `ai.py:217-222,305-314`). **Le 4ᵉ motif 503 LLM est distinct** (cf. piège 12).
3. **Épisodes/personnes non supportés par l'IA vectorielle** : ranking au niveau `movie`/`tv` uniquement (`Literal["movie","tv"]`, `ai.py:75,91`). `find_by_imdb_id` ignore défensivement `tv_episode_results`/`tv_season_results`/`person_results` (`tmdb_service.py:281`) → comptés en `resolutionFailed` et droppés.
4. **Cap 20 fetches TMDB frais par `/rank`** : `HYDRATE_CAP = 20`, timeout 10 s/tâche (`recommendation_service.py:28-29,169-176`) ; le surplus + timeouts + échecs → `cacheMissesDropped` (`stats.dropped`, `recommendation_service.py:200`). Le client re-appelle une fois le cache chaud.
5. **Le rebuild ne tourne JAMAIS au boot** : uniquement via `POST /api/ai/embed/rebuild` (`embedding_worker.py:3,132`). Idempotent : scan `embedded_at IS NULL`, curseur `tmdb_id > :cursor` (`embedding_worker.py:76`), `PAGE_SIZE=50`, DELETE-puis-INSERT sur `vec0` (UPSERT interdit sur table virtuelle, `embedding_worker.py:108-114`). Jobs en mémoire, `JOBS_CAP=100`.
6. **Migrations idempotentes** : tout DDL en `IF NOT EXISTS` / `ADD COLUMN` gardé ; nouvelle migration **en fin** de `run_migrations()`. **Ordre d'exécution : 001-007, puis 008 (sur connexion dédiée), puis 009, puis 010** (`migrations.py:20-32`). M008 (`ai_embeddings` vec0 `FLOAT[384]` + `ai_tmdb_cache`, `migrations.py:317-346`) **dépend du chargement de sqlite-vec** (`register_sqlite_vec_listener`, `database.py:25-54`). 7. **`fcntl.flock` = POSIX** : l'élection master-worker (`main.py:197,226-227`) ne fonctionne que sous Linux/Docker (pas Windows natif). Seul le master lance le scheduler ; fermer le fd libère le flock (`main.py:333-335`).
8. **SQLite « database is locked »** : passer par `commit_with_retry` / `run_with_retry` (`db_retry.py:21`, délais 0.1/0.5/1.0 s au-dessus du `busy_timeout=5000`). WAL activé. Ne jamais ouvrir plusieurs writers sans retry.
